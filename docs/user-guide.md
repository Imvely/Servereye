## 1. 소개

**ServerEye**는 Windows 및 Linux 서버를 위한 **에이전트리스(Agentless) 서버 모니터링 애플리케이션**입니다.

모니터링 대상 서버에 별도의 에이전트를 설치할 필요 없이, Windows는 WinRM, Linux는 SSH 프로토콜을 통해 원격으로 서버 상태를 수집합니다. 20~30대 규모의 서버 인프라를 관리하는 소규모 팀에 최적화되어 있습니다.

### 주요 특징

- **에이전트리스 방식**: 모니터링 대상 서버에 추가 소프트웨어 설치 불필요
- **데스크톱 애플리케이션**: 시스템 트레이에 상주하며 백그라운드에서 동작
- **실시간 모니터링**: WebSocket을 통한 실시간 메트릭 업데이트
- **다양한 메트릭**: CPU, 메모리, 디스크, 네트워크, 프로세스, 서비스, 이벤트 로그
- **알림 시스템**: 임계값 기반 알림 및 Webhook 연동 (Slack, Teams, Webex)
- **리포트 생성**: Excel 형식의 모니터링 리포트 자동 생성
- **헬스 체크**: PING, TCP, HTTP 방식의 가용성 점검

### 시스템 요구사항

| 항목 | 요구사항 |
|------|----------|
| 운영체제 | Windows 10 이상 |
| 메모리 | 4GB 이상 권장 |
| 디스크 | 500MB 이상 여유 공간 |
| 네트워크 | 모니터링 대상 서버와 통신 가능 |
| 포트 | WinRM(5985/5986), SSH(22) |

## 2. 설치 방법

### 2.1 설치 파일 다운로드

`ServerEye_Setup_v1.0.0.exe` 설치 파일을 다운로드합니다.

### 2.2 설치 진행

1. `ServerEye_Setup_v1.0.0.exe`를 실행합니다.
2. 설치 마법사의 안내에 따라 설치를 진행합니다.
3. 설치 경로를 선택하거나 기본 경로를 사용합니다.
4. 설치가 완료되면 **마침** 버튼을 클릭합니다.

### 2.3 실행 방법

설치 완료 후 다음 세 가지 방법으로 ServerEye를 실행할 수 있습니다:

- **바탕화면 바로가기**: 바탕화면의 ServerEye 아이콘을 더블 클릭
- **시작 메뉴**: 시작 메뉴에서 "ServerEye"를 검색하여 실행
- **시스템 트레이**: 이미 실행 중인 경우 시스템 트레이 아이콘을 더블 클릭

> ServerEye는 기본적으로 `http://localhost:52800` 에서 웹 UI를 제공합니다. 실행 시 자동으로 기본 브라우저에서 열립니다.

## 3. 초기 설정

### 3.1 기본 로그인 정보

ServerEye를 처음 실행하면 로그인 화면이 표시됩니다. 기본 관리자 계정으로 로그인하세요.

| 항목 | 값 |
|------|------|
| 사용자명 | `admin` |
| 비밀번호 | `admin1234` |

### 3.2 최초 실행 시 해야 할 일

#### 비밀번호 변경

보안을 위해 로그인 후 **반드시 기본 비밀번호를 변경**하세요.

1. 우측 상단의 사용자 아이콘을 클릭합니다.
2. **비밀번호 변경**을 선택합니다.
3. 현재 비밀번호(`admin1234`)를 입력하고 새 비밀번호를 설정합니다.

#### 서버 등록

모니터링할 서버를 등록합니다. 자세한 내용은 [5. 서버 관리](#5-서버-관리)를 참고하세요.

## 4. 대시보드

대시보드는 ServerEye의 메인 화면으로, 전체 서버의 상태를 한눈에 파악할 수 있습니다.

### 4.1 서버 카드

각 서버는 개별 카드로 표시되며, 다음 정보를 실시간으로 보여줍니다:

- **CPU 사용률** (%)
- **메모리 사용률** (%)
- **디스크 사용률** (%)
- **상태 배지**: 서버의 현재 상태를 색상으로 구분
  - `정상` (초록): 모든 메트릭이 정상 범위
  - `경고` (노랑): 하나 이상의 메트릭이 경고 임계값 초과
  - `위험` (빨강): 하나 이상의 메트릭이 위험 임계값 초과
  - `오프라인` (회색): 서버에 연결할 수 없음
  - `유지보수` (파랑): 유지보수 모드 활성화 중

### 4.2 실시간 업데이트

대시보드의 모든 데이터는 WebSocket을 통해 실시간으로 업데이트됩니다. 별도의 새로고침 없이 최신 상태를 확인할 수 있습니다.

### 4.3 뷰 전환

대시보드 상단에서 두 가지 뷰를 전환할 수 있습니다:

- **카드 뷰**: 시각적으로 서버 상태를 한눈에 파악 (기본값)
- **테이블 뷰**: 서버 목록을 테이블 형태로 표시하여 비교 분석에 유리

### 4.4 필터링

대시보드에서 다양한 조건으로 서버를 필터링할 수 있습니다:

- **그룹 필터**: 서버 그룹별로 필터링
- **상태 필터**: 서버 상태별로 필터링 (정상, 경고, 위험, 오프라인 등)

### 4.5 인프라 요약 카드

대시보드 상단에는 전체 인프라의 요약 정보가 표시됩니다:

- 전체 서버 수
- 정상 / 경고 / 위험 / 오프라인 서버 수
- 활성 알림 수

## 5. 서버 관리

### 5.1 서버 추가

서버 관리 페이지에서 **서버 추가** 버튼을 클릭하여 모니터링할 서버를 등록합니다.

#### 필수 입력 항목

| 항목 | 설명 |
|------|------|
| 호스트명 | 서버의 호스트 이름 |
| 표시 이름 | 대시보드에 표시될 이름 |
| IP 주소 | 서버의 IP 주소 |
| OS 유형 | `windows` 또는 `linux` 선택 |
| 접속 계정 | 원격 접속에 사용할 사용자 이름 |
| 접속 비밀번호 | 원격 접속에 사용할 비밀번호 |

#### 선택 입력 항목

| 항목 | 설명 | 기본값 |
|------|------|--------|
| 도메인 | Windows 도메인 | - |
| SSH 포트 | Linux 서버의 SSH 포트 | 22 |
| WinRM 포트 | Windows 서버의 WinRM 포트 | 5985 |
| SSL 사용 | WinRM SSL 사용 여부 | 비활성 |
| SSH 키 경로 | SSH 키 파일 경로 (비밀번호 대신 사용) | - |
| 그룹 | 서버가 속할 그룹 | 미분류 |
| 위치 | 서버 물리적 위치 | - |
| 설명 | 서버에 대한 설명 | - |
| 태그 | 서버에 부여할 태그 (콤마로 구분) | - |

### 5.2 연결 테스트

서버를 추가하기 전에 **연결 테스트** 버튼을 클릭하여 서버와의 통신이 정상인지 확인할 수 있습니다.

- Windows 서버: WinRM 프로토콜로 연결 테스트
- Linux 서버: SSH 프로토콜로 연결 테스트

연결 테스트 성공 시 해당 서버의 OS 버전, CPU, 메모리 등 기본 하드웨어 정보가 자동으로 채워집니다.

### 5.3 서버 그룹 및 태그

서버를 체계적으로 관리하기 위해 그룹과 태그를 활용할 수 있습니다.

- **그룹**: 서버를 논리적으로 분류합니다 (예: "웹서버", "DB서버", "개발서버").
- **태그**: 유연한 라벨링으로 서버를 분류합니다 (예: "production", "staging", "critical").

그룹과 태그는 대시보드 필터링 및 알림 규칙 설정에 활용됩니다.

### 5.4 유지보수 모드

서버 점검 등의 이유로 일시적으로 알림을 중지해야 할 때 유지보수 모드를 활성화할 수 있습니다.

- 유지보수 모드 중에는 해당 서버에 대한 알림이 발생하지 않습니다.
- 유지보수 종료 시각을 설정하면 해당 시각에 자동으로 유지보수 모드가 해제됩니다.
- 대시보드에서 유지보수 중인 서버는 파란색 상태 배지로 표시됩니다.

### 5.5 서버 Import / Export

다수의 서버를 일괄 등록하거나 서버 구성을 백업할 수 있습니다.

- **Export**: 현재 등록된 서버 목록을 파일로 내보내기
- **Import**: 파일에서 서버 목록을 일괄 불러오기

## 6. 서버 상세 모니터링

대시보드에서 서버 카드를 클릭하면 해당 서버의 상세 모니터링 화면으로 이동합니다. 상세 화면은 탭으로 구성되어 있습니다.

### 6.1 Overview 탭

서버의 전체 상태를 한눈에 볼 수 있는 요약 화면입니다.

- **CPU / 메모리 / 디스크 / 네트워크 요약 정보** 표시
- **주요 메트릭 차트**: 시간별 추이를 실시간 차트로 확인
- 가동 시간(Uptime) 표시
- 서버 하드웨어 정보 요약

### 6.2 CPU 탭

CPU 관련 상세 정보를 모니터링합니다.

- **현재 CPU 사용률** (%)
- **Load Average**: 1분 / 5분 / 15분 평균 부하 (Linux)
- **CPU 사용률 추이 차트**: 시간대별 CPU 사용률 변화 그래프
- 시간 범위 선택 (1시간, 6시간, 24시간, 7일 등)

### 6.3 Memory 탭

메모리 사용 현황을 상세 모니터링합니다.

- **전체 메모리 / 사용 중 / 사용률** 표시
- **Swap 메모리**: 전체 / 사용 중
- **메모리 사용률 추이 차트**: 시간대별 변화 그래프
- 시간 범위 선택

### 6.4 Disk 탭

디스크 파티션별 사용 현황 및 I/O 성능을 모니터링합니다.

- **파티션별 사용량**: 각 파티션의 전체 용량, 사용량, 사용률
- **사용률 시각화**: 파티션별 사용률 바 그래프
- **Disk I/O**: 읽기/쓰기 속도 (MB/s) 추이 차트

### 6.5 Network 탭

네트워크 트래픽 및 연결 상태를 모니터링합니다.

- **현재 네트워크 연결 수**
- **In/Out 트래픽**: 인바운드/아웃바운드 트래픽 추이 차트
- 네트워크 인터페이스별 정보

### 6.6 Service 탭

Windows 서버의 서비스 목록과 상태를 확인합니다.

- **서비스 목록**: 서비스 이름, 표시 이름, 상태, 시작 유형
- **서비스 상태**: Running, Stopped 등
- **PID 및 메모리 사용량** 표시
- 서비스 이름으로 검색 기능

> 이 탭은 Windows 서버에서만 표시됩니다.

### 6.7 Process 탭

서버에서 실행 중인 프로세스 목록을 확인합니다.

- **프로세스 목록**: PID, 이름, 사용자, CPU%, 메모리(MB), 메모리%, 스레드 수, 상태
- **정렬 기능**: 각 컬럼 헤더를 클릭하여 오름차순/내림차순 정렬
- **프로세스 수** 표시
- 프로세스 이름으로 검색 기능

### 6.8 Log 탭

서버의 이벤트 로그를 실시간으로 확인합니다.

- **로그 목록**: 발생 시각, 로그 소스, 레벨, 메시지
- **레벨 필터**: Error, Warning, Information 등 로그 레벨별 필터링
- **검색 기능**: 로그 메시지 내 키워드 검색
- **자동 스크롤**: 새로운 로그 발생 시 자동으로 하단으로 스크롤

### 6.9 Health Check 탭

서버의 가용성 점검 결과를 확인합니다.

헬스 체크는 세 가지 방식을 지원합니다:

| 유형 | 설명 |
|------|------|
| **PING** | ICMP 핑으로 서버 응답 여부 확인 |
| **TCP** | 특정 포트에 TCP 연결을 시도하여 서비스 가용성 확인 |
| **HTTP** | HTTP 요청을 보내 웹 서비스 정상 여부 확인 |

각 헬스 체크에 대해 다음 정보를 확인할 수 있습니다:
- 최근 점검 결과 (성공/실패)
- 응답 시간 (ms)
- HTTP 상태 코드 (HTTP 유형인 경우)
- 에러 메시지 (실패 시)

## 7. 알림 시스템

### 7.1 알림 규칙

알림 규칙을 설정하여 서버 메트릭이 지정된 임계값을 초과할 때 자동으로 알림을 받을 수 있습니다.

#### 알림 규칙 생성

1. **알림 규칙** 페이지에서 **규칙 추가** 버튼을 클릭합니다.
2. 다음 항목을 설정합니다:

| 항목 | 설명 |
|------|------|
| 규칙 이름 | 알림 규칙의 이름 |
| 설명 | 규칙에 대한 설명 |
| 대상 서버 | 특정 서버 또는 서버 그룹 선택 |
| 모니터링 메트릭 | CPU 사용률, 메모리 사용률, 디스크 사용률 등 |
| 경고 임계값 | 경고(Warning) 수준의 임계값 |
| 위험 임계값 | 위험(Critical) 수준의 임계값 |
| 지속 시간 | 임계값 초과가 지속되어야 하는 시간 (초) |
| 쿨다운 | 동일 알림이 반복되지 않는 최소 간격 (초) |

#### 지속 시간 (Duration)

순간적인 메트릭 스파이크로 인한 불필요한 알림을 방지합니다. 예를 들어, 지속 시간을 30초로 설정하면 CPU 사용률이 30초 이상 연속으로 임계값을 초과할 때만 알림이 발생합니다.

#### 쿨다운 (Cooldown)

동일한 규칙에 대해 알림이 너무 빈번하게 발생하는 것을 방지합니다. 기본값은 300초(5분)입니다.

### 7.2 알림 이력 및 관리

발생한 모든 알림은 **알림** 페이지에서 확인할 수 있습니다.

- **알림 목록**: 발생 시각, 서버, 심각도, 메트릭, 값, 메시지 표시
- **심각도 구분**: Warning (경고), Critical (위험)
- **시간순 정렬**: 최신 알림이 상단에 표시

### 7.3 알림 확인 및 해결

- **확인 (Acknowledge)**: 알림을 확인했음을 표시합니다. 확인한 사용자 이름과 시각이 기록됩니다.
- **해결 (Resolve)**: 문제가 해결되었음을 표시합니다. 해결 시각이 기록됩니다.

### 7.4 Webhook 알림

외부 협업 도구로 알림을 전송할 수 있습니다. 지원하는 플랫폼:

| 플랫폼 | 설명 |
|--------|------|
| **Slack** | Slack Incoming Webhook을 통한 알림 전송 |
| **Microsoft Teams** | Teams Incoming Webhook을 통한 알림 전송 |
| **Webex** | Webex Incoming Webhook을 통한 알림 전송 |

Webhook 설정은 **설정 > 외부 연동** 에서 구성합니다.

## 8. 리포트

### 8.1 리포트 생성

모니터링 데이터를 Excel 형식의 리포트로 생성할 수 있습니다.

1. **리포트** 페이지에서 **리포트 생성** 버튼을 클릭합니다.
2. 다음 항목을 설정합니다:

| 항목 | 설명 |
|------|------|
| 기간 | 리포트에 포함할 시작/종료 날짜 선택 |
| 대상 서버 | 리포트에 포함할 서버 선택 (다중 선택 가능) |
| 리포트 유형 | 리포트의 종류 선택 |

3. **생성** 버튼을 클릭하면 Excel(.xlsx) 파일이 생성됩니다.

### 8.2 리포트 내용

리포트에는 선택한 기간 동안의 다음 정보가 포함됩니다:

- 서버별 CPU / 메모리 / 디스크 사용률 통계 (평균, 최대, 최소, 95th percentile)
- 시간대별 추이 데이터
- 알림 발생 이력
- 가동 시간 및 다운타임 기록

### 8.3 다운로드 이력

생성된 리포트는 **다운로드 이력**에서 언제든 다시 다운로드할 수 있습니다.

- 리포트 이름, 유형, 기간, 파일 크기, 생성자, 생성 일시 확인 가능
- 이전에 생성한 리포트를 다시 다운로드

## 9. 설정

**설정** 페이지에서 ServerEye의 동작을 세부적으로 구성할 수 있습니다. 설정은 다음 카테고리로 구분됩니다.

### 9.1 일반 설정 (General)

애플리케이션의 전반적인 동작에 관한 설정입니다.

- 애플리케이션 이름
- 기본 언어
- 시간대 설정

### 9.2 수집 주기 (Collection Intervals)

서버 메트릭 수집 주기를 설정합니다.

- **기본 수집 간격**: 메트릭을 수집하는 기본 주기 (초 단위)
- 프로세스 정보 수집 활성화/비활성화
- 서비스 정보 수집 활성화/비활성화
- 이벤트 로그 수집 활성화/비활성화

> 서버별로 개별 수집 간격을 설정할 수도 있습니다.

### 9.3 보존 정책 (Retention Policy)

수집된 데이터의 보존 기간을 설정합니다.

- **Raw 데이터 보존 기간**: 원본 메트릭 데이터 보존 일수
- **5분 집계 데이터 보존 기간**
- **시간별 집계 데이터 보존 기간**
- **로그 보존 기간**
- **알림 이력 보존 기간**

보존 기간이 지난 데이터는 스케줄러에 의해 자동으로 삭제됩니다.

### 9.4 기본 임계값 (Default Thresholds)

새 서버를 추가할 때 기본으로 적용되는 알림 임계값을 설정합니다.

- CPU 경고/위험 임계값
- 메모리 경고/위험 임계값
- 디스크 경고/위험 임계값

### 9.5 외부 연동 (External Integrations)

외부 서비스와의 연동을 설정합니다.

#### Webhook 설정

| 항목 | 설명 |
|------|------|
| Webhook URL | 알림을 전송할 Webhook 엔드포인트 URL |
| Webhook 유형 | Slack, Teams, Webex 중 선택 |
| 활성화 여부 | Webhook 전송 활성화/비활성화 |

설정 완료 후 **테스트 전송** 버튼으로 정상 작동 여부를 확인할 수 있습니다.

## 10. 사용자 관리

ServerEye는 역할 기반 접근 제어(RBAC)를 지원합니다.

### 10.1 사용자 역할

| 역할 | 설명 | 권한 |
|------|------|------|
| **Admin** (관리자) | 시스템 전체 관리 | 모든 기능 접근 가능 (설정, 사용자 관리 포함) |
| **Operator** (운영자) | 서버 운영 담당 | 서버 관리, 알림 확인/해결, 리포트 생성 가능 |
| **Viewer** (조회자) | 읽기 전용 | 대시보드, 서버 상세 정보, 알림 조회만 가능 |

### 10.2 사용자 생성

1. **사용자 관리** 페이지에서 **사용자 추가** 버튼을 클릭합니다.
2. 다음 정보를 입력합니다:
   - 사용자명
   - 표시 이름
   - 비밀번호
   - 역할 (Admin / Operator / Viewer)
3. **저장** 버튼을 클릭합니다.

### 10.3 사용자 편집

기존 사용자의 정보를 수정할 수 있습니다:

- 표시 이름 변경
- 역할 변경
- 비밀번호 재설정

### 10.4 사용자 비활성화

사용자를 삭제하는 대신 **비활성화**하여 로그인을 차단할 수 있습니다. 비활성화된 사용자는 로그인할 수 없지만 기록은 유지됩니다.

- 클릭할 때마다 라이트 모드와 다크 모드가 전환됩니다.

## 12. 시스템 트레이

ServerEye는 시스템 트레이(알림 영역)에 상주하며 백그라운드에서 모니터링을 수행합니다.

### 12.1 트레이 아이콘 색상

트레이 아이콘의 색상은 전체 서버의 상태를 나타냅니다:

| 아이콘 색상 | 의미 |
|-------------|------|
| 초록 | 모든 서버 정상 |
| 노랑 | 경고 상태의 서버 존재 |
| 빨강 | 위험 상태 또는 오프라인 서버 존재 |

### 12.2 우클릭 메뉴

시스템 트레이 아이콘을 **우클릭**하면 다음 메뉴가 표시됩니다:

- **열기**: 브라우저에서 ServerEye 웹 UI를 엽니다.
- **종료**: ServerEye를 완전히 종료합니다.

### 12.3 더블 클릭

시스템 트레이 아이콘을 **더블 클릭**하면 기본 브라우저에서 ServerEye 웹 UI가 열립니다.

## 13. FAQ / 문제 해결

### Q1. 서버 연결 테스트에 실패합니다.

**Windows 서버 (WinRM)**

- 대상 서버에서 WinRM 서비스가 활성화되어 있는지 확인하세요.
  ```powershell
  winrm quickconfig
  ```
- 방화벽에서 WinRM 포트(기본 5985)가 열려 있는지 확인하세요.
- 접속 계정에 원격 관리 권한이 있는지 확인하세요.
- WinRM SSL을 사용하는 경우 포트를 5986으로 설정하고 SSL 옵션을 활성화하세요.

**Linux 서버 (SSH)**

- 대상 서버에서 SSH 서비스가 실행 중인지 확인하세요.
  ```bash
  systemctl status sshd
  ```
- 방화벽에서 SSH 포트(기본 22)가 열려 있는지 확인하세요.
- 접속 계정의 사용자명과 비밀번호(또는 SSH 키)가 올바른지 확인하세요.

### Q2. 대시보드에 서버 데이터가 표시되지 않습니다.

- 서버의 상태가 **오프라인**인지 확인하세요. 오프라인인 경우 연결 설정을 점검하세요.
- 서버를 방금 추가한 경우 첫 번째 수집이 완료될 때까지 잠시 기다려 주세요.
- 브라우저를 새로고침하거나 WebSocket 연결 상태를 확인하세요.

### Q3. 알림이 발생하지 않습니다.

- **알림 규칙** 페이지에서 해당 규칙이 **활성화** 상태인지 확인하세요.
- 대상 서버가 **유지보수 모드**가 아닌지 확인하세요.
- 쿨다운 시간 내에 이미 알림이 발생한 적이 있는지 확인하세요.
- 지속 시간 설정을 확인하세요. 순간적인 스파이크는 알림을 발생시키지 않습니다.

### Q4. Webhook 알림이 전송되지 않습니다.

- **설정 > 외부 연동**에서 Webhook URL이 올바르게 입력되었는지 확인하세요.
- Webhook이 **활성화** 상태인지 확인하세요.
- **테스트 전송** 버튼으로 연결을 확인하세요.
- ServerEye가 실행 중인 PC에서 Webhook URL로 네트워크 접근이 가능한지 확인하세요.

### Q5. 리포트 생성에 실패합니다.

- 선택한 기간 내에 수집된 데이터가 있는지 확인하세요.
- 디스크 여유 공간이 충분한지 확인하세요.
- 리포트 저장 경로에 쓰기 권한이 있는지 확인하세요.

### Q6. 프로그램이 시작되지 않거나 포트 충돌이 발생합니다.

- 기본 포트(52800)가 다른 프로그램에서 사용 중인지 확인하세요.
  ```powershell
  netstat -ano | findstr :52800
  ```
- 이미 ServerEye가 실행 중인지 시스템 트레이를 확인하세요.
- 관리자 권한으로 실행해 보세요.

### Q7. 비밀번호를 잊어버렸습니다.

- Admin 권한을 가진 다른 사용자에게 비밀번호 재설정을 요청하세요.
- 모든 Admin 계정의 비밀번호를 잊어버린 경우, 데이터 디렉토리(`%LOCALAPPDATA%\ServerEye`)의 데이터베이스 파일을 통해 초기화해야 합니다.

### Q8. 다크 모드가 적용되지 않습니다.

- 브라우저의 캐시를 삭제한 후 다시 시도해 보세요.
- 지원되는 브라우저(Chrome, Edge, Firefox 최신 버전)를 사용하고 있는지 확인하세요.
